Module 1 • Q1

Explain the three-schema architecture of DBMS with a neat diagram. How does it provide logical and physical data independence?

1. Definition and Introduction

The three-schema architecture (also referred to as the ANSI-SPARC architecture) is a foundational framework for database management systems designed to separate the user applications from the physical database. Its primary objective is to achieve data abstraction and data independence. By dividing the database into three levels, it ensures that users are shielded from the complex internal storage details, while administrators can manage data effectively.

Figure 1.1: Three-Schema Architecture of DBMS

2. Detailed Explanation of the Three Levels

  • Internal Level (Physical Level): This is the lowest level of abstraction. It defines how data is physically stored on storage devices. It deals with file organizations, access paths, B+ tree indexing, hashing techniques, and data compression. The internal schema describes the complete physical structure of the database.
  • Conceptual Level (Logical Level): This middle level describes the logical structure of the entire database for a community of users. It hides physical storage details and concentrates on defining entities, attributes, relationships, user operations, and constraints. It acts as the backbone integrating all external views.
  • External Level (View Level): This is the highest level, presenting data to individual users or applications. A database can have multiple external schemas. Each schema describes the part of the database that a specific user group is interested in, hiding the rest for security and simplicity.

3. Database Mappings

To retrieve or update data, the DBMS must transform requests and results between these levels. This process is called mapping.

  1. External-Conceptual Mapping: Maps a specific user view to the conceptual schema. It translates user queries into conceptual queries.
  2. Conceptual-Internal Mapping: Maps the logical schema to the physical storage. It dictates exactly where and how to find the logical records on the disk.

4. Achieving Data Independence

The architecture inherently supports data independence—the capacity to change the schema at one level without having to change the schema at the next higher level.

  • Logical Data Independence: Modifying the conceptual schema (e.g., adding a new attribute or splitting a table) does not affect the external schemas or application programs. The mapping between external and conceptual layers absorbs the changes.
  • Physical Data Independence: Modifying the internal schema (e.g., creating a new index or changing from a heap file to a hash file) does not affect the conceptual schema. The mapping between conceptual and internal layers handles the differences.

5. Advantages and Disadvantages

Advantages:
  • High security (users see only relevant data).
  • Data independence ensures applications are resilient to structural changes.
  • Simplified user interaction.
  • Modular database administration.
Disadvantages:
  • High compilation overhead due to two levels of mapping.
  • Increased complexity in DBMS software development.
  • Potential performance degradation compared to direct file access.
Summary: The three-schema architecture is an essential framework in modern DBMS that decouples applications from physical storage, providing high security, logical abstraction, and physical data independence.
Module 1 • Q2

Define data independence. Explain logical data independence and physical data independence with suitable examples.

1. Definition of Data Independence

Data independence refers to the characteristic of a database system that allows the schema at one level of the database system to be modified without requiring changes to the schema at the next higher level. This is a critical feature distinguishing a robust DBMS from primitive file-based systems.

2. Types of Data Independence

Based on the three-schema architecture, data independence is broadly classified into two categories:

2.1 Physical Data Independence

Physical data independence is the ability to alter the internal schema (physical storage structures) without causing any changes to the conceptual schema or external schemas.

  • Purpose: It is generally achieved to optimize the performance of the database (e.g., faster retrieval or more efficient storage).
  • Mechanism: The conceptual-internal mapping is adjusted by the DBMS internally so that logical queries remain identical.
  • Examples:
    1. Changing the primary file organization from sequential to a B+ tree indexed file.
    2. Using a different hashing algorithm to store records.
    3. Moving data files from an SSD to a high-capacity HDD.
    4. Altering the block size or compression algorithms.
  • Impact: Application programs (e.g., a query like SELECT * FROM Employee) run seamlessly without any code changes.

2.2 Logical Data Independence

Logical data independence is the ability to modify the conceptual schema (logical structure) without forcing changes to the external schemas or existing application programs.

  • Purpose: It handles the logical evolution of the database as business requirements change over time.
  • Mechanism: The external-conceptual mapping is updated. Views are redefined to produce the same virtual table structure expected by legacy applications.
  • Examples:
    1. Adding a new attribute like DateOfBirth to the STUDENT table. Existing applications that only query USN and Name are unaffected.
    2. Splitting a single EMPLOYEE(ID, Name, DeptName, Location) relation into two normalized relations: EMPLOYEE(ID, Name, DeptID) and DEPARTMENT(DeptID, DeptName, Location). A view can be created to join these tables, keeping older applications functional.
    3. Removing an attribute that is no longer used by any active application.

3. Tabular Comparison for Quick Revision

Parameter Physical Data Independence Logical Data Independence
Definition Immunity of conceptual schema to changes in internal schema. Immunity of external schema to changes in conceptual schema.
Complexity Relatively easier to implement as it deals with storage mechanics. Highly complex to achieve due to strict logical dependencies and constraints.
Primary Reason for Change To improve query performance or optimize storage space. To adapt to changing business requirements or database restructuring.
Level of Mapping Modified Conceptual-Internal mapping. External-Conceptual mapping.
Conclusion: Data independence ensures that the database remains scalable, maintainable, and adaptable to future technological and business changes without rewriting costly application software. Physical independence optimizes performance, while logical independence facilitates structural evolution.

What is an Entity-Relationship (ER) model? Draw a neat ER diagram for a Library Management System and explain its components.

1. Definition and Introduction to ER Model

The Entity-Relationship (ER) model, introduced by Peter Chen in 1976, is a high-level conceptual data model used extensively in the initial phases of database design. It visually maps the real-world domain by viewing the system as a collection of basic objects called entities, their characteristics called attributes, and the logical associations between them called relationships.

2. Core Components of the ER Model

  1. Entity: A "thing" or "object" in the real world with an independent existence. Modeled as a rectangle.
  2. Attribute: Properties or characteristics that describe an entity. Modeled as an ellipse.
  3. Relationship: An association connecting two or more entities. Modeled as a diamond.
  4. Key Attribute: An attribute that uniquely identifies an entity instance. Modeled as an ellipse with underlined text.
  5. Lines/Links: Connect attributes to entities and entities to relationships, defining participation.

3. Neat ER Diagram: Library Management System

Figure 1.3: ER Diagram for a University Library Management System

4. Step-by-Step Explanation of the ER Diagram Components

  1. Entities:
    • BOOK: Stores general information about a book title.
    • COPY: Represents physical books residing on library shelves.
    • MEMBER: Represents registered users (students/staff) who can borrow books.
  2. Attributes:
    • BOOK has ISBN (Primary Key), Title, and Author.
    • COPY has Copy_ID (Primary Key), Status (Available/Issued), and Rack_No.
    • MEMBER has Member_ID (Primary Key), Name, and Phone.
  3. Relationships & Cardinalities:
    • HAS (BOOK ↔ COPY): Cardinality is 1:N. One specific book (e.g., "Fundamentals of DBMS") can have multiple physical copies in the library. A single physical copy belongs to exactly one book title.
    • BORROWS (MEMBER ↔ COPY): Cardinality is M:N. Over time, a member can borrow multiple copies, and a specific copy can be borrowed by multiple members.
    • Relationship Attributes: The BORROWS relationship itself generates data. When a member borrows a copy, it records the Issue_Date and Due_Date. These belong to the relationship, not to the entity independently.

5. Design Advantages

  • Granular Tracking: By splitting Book and Copy into two entities, the library can independently track the condition and status of every physical textbook.
  • Historical Accuracy: M:N borrowing with relationship attributes allows the system to construct a precise transaction log.
Conclusion: An ER Diagram offers a highly structured conceptual visualization of complex systems. The Library ER design meticulously segregates bibliographic data, physical inventory, user profiles, and transactional data into robust constraints.
Module 1 • Q4

Explain different types of attributes, relationships, and constraints used in ER modeling with examples.

1. Introduction to ER Constructs

The Entity-Relationship modeling language uses three primary grammatical elements to formulate database structures: Attributes (properties), Relationships (verbs/actions), and Constraints (rules).

2. Types of Attributes

An attribute provides descriptive metadata for an entity type.

  1. Simple (Atomic) Attribute: An attribute that cannot be further subdivided into smaller meaningful components.
    Example: Gender or Age of an employee.
  2. Composite Attribute: An attribute composed of several basic attributes, forming a hierarchy. Modeled as an ellipse connected to other ellipses.
    Example: Name can be divided into First_Name, Middle_Name, and Last_Name. Address can be divided into Street, City, State, and ZIP.
  3. Single-valued Attribute: Takes only one value for a particular entity instance.
    Example: Date_of_Birth. A person has only one valid date of birth.
  4. Multi-valued Attribute: Can have a set of multiple values for a single entity. Modeled as a double-ellipse.
    Example: Phone_Numbers (a person can have home, work, and mobile numbers), or Degrees_Obtained.
  5. Derived Attribute: An attribute whose value is dynamically calculated or derived from other related attributes or entities. Modeled as a dashed ellipse.
    Example: Age is derived dynamically from Date_of_Birth and the current system date.
  6. Key Attribute: An attribute or a combination of attributes that uniquely distinguishes every entity within the entity set. Modeled with an underline.
    Example: USN in the Student entity.

3. Types of Relationships (Degree of Relationship)

The degree of a relationship refers to the number of participating entity types.

  1. Unary (Recursive) Relationship: A relationship where an entity associates with itself.
    Example: An EMPLOYEE entity has a relationship Supervises with another EMPLOYEE (Managers supervising subordinates).
  2. Binary Relationship: A relationship involving exactly two entity types. This is the most common degree.
    Example: STUDENT Enrolls_In COURSE.
  3. Ternary Relationship: A simultaneous relationship among three entity types.
    Example: A DOCTOR prescribes a DRUG to a PATIENT.

4. ER Constraints (Structural Rules)

Constraints restrict the data to ensure database accuracy, integrity, and real-world semantic adherence.

A. Cardinality Ratio Constraints (for Binary Relationships)

Specifies the maximum number of relationship instances an entity can participate in.

  • 1:1 (One-to-One): An employee manages exactly one department, and a department has exactly one manager.
  • 1:N (One-to-Many): A department employs many employees, but an employee belongs to exactly one department.
  • M:N (Many-to-Many): A student can enroll in multiple courses, and a course can accommodate multiple students.
B. Participation Constraints (Min-Max Constraints)

Specifies whether the existence of an entity depends on its being related to another entity.

  • Total Participation (Existence Dependency): Every entity must participate in the relationship. Modeled by a double line.
    Example: Every EMPLOYEE must belong to a DEPARTMENT.
  • Partial Participation: Some entities may participate, but it is not mandatory. Modeled by a single line.
    Example: Not every EMPLOYEE manages a department. Hence, the participation of Employee in the "Manages" relationship is partial.
Conclusion: By accurately combining atomic, composite, and multi-valued attributes with strict cardinality and participation constraints, an ER model enforces robust data validation directly at the conceptual design phase.
Module 1 • Q5

Compare the traditional file processing system with a database management system. Discuss the advantages of DBMS over file systems.

1. Introduction: The Evolution of Data Management

In the early days of computing, commercial applications stored data using standard operating system files. Each application (e.g., Payroll, Registration) had its own set of dedicated files (flat files). This is called the Traditional File Processing System. As data grew exponentially in volume and complexity, it became unmanageable, leading to the development of centralized Database Management Systems (DBMS).

2. Point-by-Point Comparison

Parameter / Feature Traditional File Processing System Database Management System (DBMS)
Data Integration Data is isolated across discrete files, often in different formats. Data is integrated and centrally pooled into a single structured repository.
Data Redundancy Extremely high. The same data (e.g., student address) is duplicated across department and library files. Minimized. Data is normalized and stored once, eliminating duplicates.
Data Independence None. Application code is strictly tied to the physical structure of the file. Changing file formats requires rewriting code. High logical and physical data independence. Applications do not care how data is stored.
Data Consistency Poor. Updating an address in one file but forgetting to update it in another leads to data inconsistency. High. Updates propagate accurately through centralized constraints and cascading rules.
Concurrency Control Very limited. Concurrent access can lead to overwritten data or dirty reads. Robust concurrency control mechanisms (e.g., locking) allow thousands of users to read/write safely.
Security & Access Enforced by basic OS-level directory permissions. Granular, role-based authorization protocols. Can restrict access down to specific rows or columns.
Crash Recovery Manual backup required. If the system crashes mid-write, files get corrupted. Automated transaction logs (ACID properties) ensure database state is restored cleanly after crashes.

3. Core Limitations of the File Processing System

  • Data Isolation: Generating a report that requires data from two different systems (e.g., HR and Accounting) requires writing complex, custom ad-hoc programs.
  • Integrity Problems: Enforcing constraints (like "Balance cannot be less than zero") must be hardcoded into every single application. If developers make a mistake, bad data enters the system.
  • Atomicity Problems: If a money transfer operation crashes after debiting an account but before crediting the other, the file system leaves the data in an inconsistent state.

4. Prominent Advantages of DBMS

  1. Declarative Querying (SQL): Users can extract complex insights instantly using simple SQL queries without writing procedural code in C or Java.
  2. Centralized Administration: Database Administrators (DBAs) can optimize performance, manage backups, and patch security vulnerabilities from a single pane of glass.
  3. ACID Transactions: Ensures Atomicity, Consistency, Isolation, and Durability, making DBMS essential for banking, e-commerce, and enterprise applications.
  4. Data Integrity: Primary keys, foreign keys, and domain checks are built deep into the DBMS kernel, rejecting illegal data automatically.

5. Disadvantages of DBMS

  • High Initial Cost: Procuring enterprise DBMS software (like Oracle) and robust hardware clusters is expensive.
  • Complexity: Requires highly trained personnel (DBAs) to manage and tune the system.
  • Overkill for Trivial Data: For a simple standalone application handling less than a few megabytes of data, a DBMS introduces unnecessary overhead.
Conclusion: While file processing systems are simple and cheap, they fail disastrously at scale. A DBMS provides the critical abstraction, security, concurrency, and integrity necessary to manage modern, mission-critical organizational data.

Explain the fundamental operations of Relational Algebra with suitable examples.

Figure 2.1: Basic Relational Algebra Operations

1. Definition and Introduction

Relational Algebra is a formal procedural query language associated with the relational model. It consists of a set of operations that take one or two relations as input and produce a new relation as their output. This mathematical framework forms the theoretical foundation for SQL query processing and optimization in RDBMS.

2. The Six Fundamental Operations

There are six basic operations. Other operations (like Join, Intersection, Division) can be derived from these six.

A. Selection (σ)

The Selection operation filters rows (tuples) from a relation that satisfy a specific propositional formula (condition). It acts horizontally.

Notation: σ_{predicate}(R) Meaning: { t | t ∈ R and p(t) is true }

Example: Find all students in the 'CSE' department.

σ_{Dept='CSE'}(STUDENT)
B. Projection (π)

The Projection operation extracts specific columns (attributes) from a relation, discarding the rest. It automatically eliminates duplicate tuples in its result set. It acts vertically.

Notation: π_{A1, A2, ..., An}(R) Meaning: Extracts columns A1, A2 from R

Example: Display only the names and phone numbers of students.

π_{Name, Phone}(STUDENT)
C. Union (∪)

The Union operation combines tuples from two relations into a single relation. The two relations must be union-compatible (same number of attributes, and corresponding domains must be compatible).

Notation: R ∪ S Meaning: { t | t ∈ R or t ∈ S }

Example: Get all active students and alumni.

STUDENT ∪ ALUMNI
D. Set Difference (−)

The Set Difference operation returns tuples that are present in relation R but not in relation S. Relations must be union-compatible.

Notation: R − S Meaning: { t | t ∈ R and t ∉ S }

Example: Find students who have registered for 'DBMS' but not for 'OS'.

DBMS_STUDENTS − OS_STUDENTS
E. Cartesian Product (×)

Also known as Cross Product, it combines information from any two relations regardless of their structure. Every tuple of R is paired with every tuple of S.

Notation: R × S Degree: Attributes of R + Attributes of S Cardinality: Tuples of R * Tuples of S

Example: Combine Employee and Department records before a selection is applied.

EMPLOYEE × DEPARTMENT
F. Rename (ρ)

The Rename operation allows us to change the name of a relation or its attributes. This is highly useful in complex queries involving self-joins.

Notation: ρ_{S(B1, B2)}(R) Meaning: Renames relation R to S, with attributes B1, B2.

3. Advantages and Applications

  • Mathematical Rigor: Provides a strict algebraic foundation preventing ambiguity.
  • Query Optimization: The DBMS optimizer converts SQL into relational algebra trees and applies equivalence rules (e.g., pushing selections down) to execute queries faster.
Conclusion: Relational algebra uses operators like selection, projection, and set operations to manipulate data relations mathematically. Understanding these operations is essential for mastering how SQL queries are processed under the hood.

Define Super Key, Candidate Key, Primary Key, Alternate Key, and Foreign Key. Explain each with suitable examples.

1. Introduction to Keys

In a relational database model, Keys are vital elements used to identify rows uniquely, enforce integrity rules, and establish logical relationships between multiple tables.

Base Example Schema: EMPLOYEE(EmpID, SSN, Passport_No, Name, DeptID)

2. Definitions and Detailed Examples

A. Super Key

Definition: A super key is any set of one or more attributes whose values collectively identify each tuple (row) uniquely within a relation.

Characteristics: A super key may contain unnecessary or redundant attributes.

Examples from Schema:

  • {EmpID}
  • {SSN}
  • {EmpID, Name}
  • {SSN, Passport_No, DeptID}

All of the above guarantee unique identification, hence they are super keys.

B. Candidate Key

Definition: A candidate key is a minimal super key. It is a super key from which no attribute can be removed without losing its uniqueness property.

Characteristics: Every candidate key is a super key, but not every super key is a candidate key. They cannot contain NULL values.

Examples from Schema:

  • {EmpID} is a candidate key.
  • {SSN} is a candidate key.
  • {Passport_No} is a candidate key.
  • (Note: {EmpID, Name} is not a candidate key because removing Name still leaves a unique key).
C. Primary Key

Definition: A primary key is a specific candidate key chosen by the database designer to uniquely identify tuples in the table.

Characteristics: There can be only one primary key per table. It enforces the Entity Integrity Constraint (it cannot be NULL).

Example from Schema: The designer explicitly selects EmpID as the Primary Key.

D. Alternate Key (Secondary Key)

Definition: All candidate keys that are not chosen as the primary key are called alternate keys.

Characteristics: They still possess unique properties and are often enforced via UNIQUE constraints in SQL.

Example from Schema: Since EmpID is the primary key, SSN and Passport_No become alternate keys.

E. Foreign Key

Definition: A foreign key is an attribute (or set of attributes) in one relation that references the primary key of another relation.

Characteristics: It establishes a link between two tables and enforces the Referential Integrity Constraint. Its domain must match the referenced primary key.

Example from Schema: If there is a table DEPARTMENT(DeptID, DeptName), then DeptID in the EMPLOYEE table is a foreign key referencing the DEPARTMENT table.

3. Summary Mapping

Conclusion: Proper identification of super, candidate, and primary keys is mandatory for normalization and entity integrity, while foreign keys are indispensable for linking normalized tables without violating referential integrity.
Module 2 • Q3

Explain the relational data model and its integrity constraints with examples.

1. Concept of the Relational Data Model

The Relational Data Model, proposed by E.F. Codd in 1970, represents the database as a collection of relations. Structurally, a relation resembles a two-dimensional table of values.

  • Relation: A mathematical term for a table.
  • Tuple: A row in the table representing a single entity instance.
  • Attribute: A column in the table representing a property of the entity.
  • Domain: The set of permissible atomic values for an attribute.
  • Degree: The total number of attributes (columns) in a relation.
  • Cardinality: The total number of tuples (rows) in a relation.

2. Relational Integrity Constraints

Integrity constraints are strict rules enforced by the DBMS to guarantee that the data remains accurate, reliable, and consistent during operations (INSERT, UPDATE, DELETE).

A. Domain Constraint

Rule: Every attribute value in a tuple must be an atomic (indivisible) value drawn directly from its defined mathematical domain.

Explanation: If an attribute is defined as INTEGER, it cannot contain strings. Similarly, data types, lengths, and nullability fall under this.

Example: In EMPLOYEE(Age), inserting the string "Twenty" violates the domain constraint. Enforcing Age > 18 via a CHECK clause is also a domain/semantic constraint.

B. Key Constraint

Rule: No two tuples in a relation can have identical values for all their attributes, and specifically, the values of the Candidate Key(s) must be strictly unique across all tuples.

Explanation: Ensures that every record can be distinctly identified.

Example: If Account_Number is a key, trying to insert a new row with an existing account number throws a primary/unique key violation error.

C. Entity Integrity Constraint

Rule: No primary key attribute (or component of a composite primary key) may contain a NULL value.

Explanation: The primary key is used to identify individual tuples in a relation. If it is null, the system cannot distinguish that tuple from others.

Example: In STUDENT(USN, Name), inserting a tuple (NULL, "John") is illegal.

D. Referential Integrity Constraint

Rule: A foreign key value in a referencing relation must either match a primary key value in the referenced relation, or be NULL (if permitted).

Explanation: It prevents "orphaned" records, ensuring relationships between tables remain valid.

Example: In EMPLOYEE(EmpID, DeptID) referencing DEPARTMENT(DeptID), if a user attempts to insert an employee with DeptID = 99, but department 99 does not exist in the DEPARTMENT table, the system will block the transaction.

3. Advantages

  • Mathematical Soundness: Backed by set theory and first-order predicate logic.
  • Data Independence: Physical data storage can change independently of the logical tabular view.
  • Simplicity: Tabular representation is intuitive for developers and end users.
Conclusion: The relational model translates chaotic data into mathematically rigorous tables. Its true power, however, lies in its integrity constraints (Domain, Key, Entity, and Referential) which form an unbreakable shield guarding the database against anomalies.
Module 2 • Q4

Explain the process of mapping an ER model into a relational schema with suitable examples.

1. Introduction

ER-to-Relational mapping is the algorithmic process of transforming a conceptual ER diagram into a logical relational schema consisting of tables, primary keys, and foreign keys.

2. Step-by-Step Mapping Rules and Algorithms

Step 1: Mapping of Regular (Strong) Entity Types
  • Rule: For each strong entity type E in the ER schema, create a relation R that includes all the simple attributes of E.
  • Key: Choose one of the key attributes of E as the primary key for R.
  • Example:
    ER: Entity STUDENT with attributes USN, Name, Age.
    Mapping: STUDENT(USN, Name, Age)
Step 2: Mapping of Weak Entity Types
  • Rule: Create a relation R for the weak entity. Include all simple attributes. Then, include the primary key of the owner (identifying) entity as a foreign key in R.
  • Key: The primary key of R is the combination of the owner's primary key and the weak entity's partial key.
  • Example:
    ER: Weak Entity DEPENDENT owned by EMPLOYEE. Partial key is Dep_Name.
    Mapping: DEPENDENT(Emp_SSN, Dep_Name, Gender, Birth_Date)
Step 3: Mapping of Binary 1:1 Relationships
  • Rule: Identify the relations S and T corresponding to the entities. Choose one (preferably the one with total participation) and include the primary key of the other as a foreign key in it.
  • Example:
    ER: 1:1 relation MANAGES between EMPLOYEE and DEPARTMENT (Total participation on Department side).
    Mapping: Add Mgr_SSN to DEPARTMENT table.
    DEPARTMENT(Dept_No, Dept_Name, Mgr_SSN)
Step 4: Mapping of Binary 1:N (One-to-Many) Relationships
  • Rule: Identify the relation S representing the "1" side and T representing the "N" side. Include the primary key of S as a foreign key in T.
  • Example:
    ER: DEPARTMENT (1) WORKS_FOR EMPLOYEE (N).
    Mapping: Add Dept_No as foreign key in EMPLOYEE.
    EMPLOYEE(Emp_SSN, Name, Dept_No)
Step 5: Mapping of Binary M:N (Many-to-Many) Relationships
  • Rule: Create a entirely new relation S to represent the relationship. Include the primary keys of both participating entities as foreign keys in S.
  • Key: The primary key of S is the combination of both foreign keys.
  • Example:
    ER: STUDENT (M) ENROLLS in COURSE (N) with relationship attribute Grade.
    Mapping: ENROLLS(USN, Course_ID, Grade)
Step 6: Mapping of Multivalued Attributes
  • Rule: Create a new relation R containing the multivalued attribute. Include the primary key of the parent entity as a foreign key.
  • Key: Primary key is the combination of the foreign key and the attribute itself.
  • Example:
    ER: EMPLOYEE has multivalued attribute Locations.
    Mapping: EMP_LOCATIONS(Emp_SSN, Location)

3. Advantages of Algorithmic Mapping

Following strict mapping rules prevents logical design anomalies, guarantees normalization (usually generating 3NF tables immediately), and seamlessly enforces ER cardinality constraints via foreign keys.

Conclusion: ER-to-Relational mapping translates conceptual shapes (rectangles, diamonds) into logical rules. Strong entities become tables, 1:N relations become foreign keys, M:N relations generate junction tables, and multivalued attributes spawn new tables.

Explain the SQL language categories DDL, DML, DCL, and TCL with suitable commands and examples.

1. Introduction to SQL

SQL (Structured Query Language) is the standard language for relational database management systems. It is comprehensive, acting as a data definition, data manipulation, and data control language. Based on their functionality, SQL commands are classified into four major categories: DDL, DML, DCL, and TCL.

2. DDL – Data Definition Language

DDL commands are used to define, alter, and drop the structure of database objects like tables, schemas, views, and indexes. DDL operations are auto-committed in most DBMS.

  • CREATE: Constructs a new database or table. 
    CREATE TABLE Employee (
  EmpID INT PRIMARY KEY,
  Name VARCHAR(50) NOT NULL,
  Salary DECIMAL(10, 2)
);
  • ALTER: Modifies an existing database object (e.g., adding a column). 
    ALTER TABLE Employee ADD DeptID INT;
  • DROP: Deletes the entire table structure and its data permanently. 
    DROP TABLE Employee;
  • TRUNCATE: Quickly removes all rows from a table while keeping the structure intact. 
    TRUNCATE TABLE Employee;

3. DML – Data Manipulation Language

DML commands are used to manage and manipulate the actual data present within the tables. These operations can usually be rolled back.

  • INSERT: Adds new rows of data into a table. 
    INSERT INTO Employee (EmpID, Name, Salary) 
VALUES (101, 'Alice', 75000);
  • UPDATE: Modifies existing data within the table based on a condition. 
    UPDATE Employee SET Salary = 80000 WHERE EmpID = 101;
  • DELETE: Removes specific rows from a table. 
    DELETE FROM Employee WHERE EmpID = 101;
  • SELECT (Sometimes classified as DQL): Retrieves data from the database. 
    SELECT Name, Salary FROM Employee WHERE Salary > 50000;

4. DCL – Data Control Language

DCL commands manage the rights, permissions, and security of the database.

  • GRANT: Gives user specific privileges. 
    GRANT SELECT, INSERT ON Employee TO 'db_user';
  • REVOKE: Withdraws privileges given previously. 
    REVOKE INSERT ON Employee FROM 'db_user';

5. TCL – Transaction Control Language

TCL commands manage the transactions (a logical unit of work) in the database, ensuring ACID properties.

  • COMMIT: Permanently saves all changes made during the current transaction. 
    COMMIT;
  • ROLLBACK: Undoes changes made in the current transaction. 
    ROLLBACK;
  • SAVEPOINT: Sets a point within a transaction to which you can later roll back. 
    SAVEPOINT sp1;
ROLLBACK TO sp1;
Conclusion: SQL's robust categorization allows administrators to build schema (DDL), developers to manipulate data (DML), security engineers to assign roles (DCL), and the system to maintain consistency (TCL).

1. Definition of a View

A View in SQL is a virtual table whose contents are defined by a query. It does not store data physically on the disk (except for materialized views); instead, it dynamically fetches data from the underlying base tables whenever queried.

2. Advantages of Views

  • Security: Hides sensitive columns from users (e.g., hiding Salary from an HR view).
  • Simplicity: Encapsulates complex joins and calculations into a single pseudo-table.
  • Logical Data Independence: Applications querying the view remain unaffected if the base table structure changes.

3. Creating a View

Syntax: CREATE VIEW view_name AS SELECT ...

Example: Create a view showing only CS students with their marks.

CREATE VIEW CS_Student_Performance AS

FROM Student S
JOIN Enroll E ON S.USN = E.USN
WHERE S.DeptID = 'CS';

You can now query it just like a table: SELECT * FROM CS_Student_Performance;

4. Updating Data Through a View (Updatable Views)

You can perform INSERT, UPDATE, and DELETE operations on a view, but these operations ultimately modify the underlying base table. However, a view is updatable ONLY IF:

  • It contains only one base table.
  • It does not contain aggregate functions, GROUP BY, or DISTINCT.

Example:

CREATE VIEW Basic_Student_View AS SELECT USN, Name FROM Student;

-- This update will modify the base 'Student' table
UPDATE Basic_Student_View SET Name = 'Bob' WHERE USN = '123';

5. Dropping a View

When a view is no longer needed, it can be deleted using the DROP VIEW command. This does not delete any data from the base tables.

DROP VIEW CS_Student_Performance;
Conclusion: Views act as secure, simplified windows into complex databases. They enforce security and simplify application code by treating complex queries as standalone virtual tables.

  • Complex Validations: Enforcing business rules that are too complex for CHECK constraints.
  • Derived Columns: Automatically calculating a Total_Price when a row is inserted.

CREATE TRIGGER Salary_Audit
AFTER UPDATE OF Salary ON EMPLOYEE
FOR EACH ROW
BEGIN
    INSERT INTO AUDIT_LOG (EmpID, Old_Salary, New_Salary, Change_Date)

Module 4 • Q1

What is Functional Dependency? Explain trivial and non-trivial functional dependencies with examples.

1. Definition of Functional Dependency (FD)

A Functional Dependency is a constraint between two sets of attributes in a relation from a database. It describes how the value of one attribute determines the value of another.

Notation: X → Y Read as: "X functionally determines Y" or "Y is functionally dependent on X"

Formal Definition: Let R be a relation schema and X and Y be sets of attributes in R. The FD X → Y holds if, for any two tuples t1 and t2 in R, if t1[X] = t2[X], then it must also be true that t1[Y] = t2[Y].

Example: In STUDENT(USN, Name, DateOfBirth), the FD USN → Name holds because if two records have the same USN, they MUST have the same Name.

2. Types of Functional Dependencies

A. Trivial Functional Dependency

An FD X → Y is trivial if the set of attributes Y is a subset of the set of attributes X (i.e., Y ⊆ X).

  • Meaning: It is always true by definition and provides no useful information for database design.
  • Examples:
    • {USN, Name} → {USN}
    • {EmpID} → {EmpID}
B. Non-Trivial Functional Dependency

An FD X → Y is non-trivial if Y is NOT a subset of X (i.e., Y ⊈ X).

  • Meaning: These are the crucial dependencies used in normalization because they define actual business rules.
  • Examples:
    • {USN} → {Name}
    • {DeptID} → {DeptName}
C. Completely Non-Trivial Functional Dependency

An FD X → Y is completely non-trivial if X and Y have no attributes in common (i.e., X ∩ Y = ∅).

  • Example: {EmpID} → {Salary}

3. Why are FDs Important?

  • They help in identifying the Candidate Keys of a relation.
  • They are the primary mathematical tool used to detect redundancies.
  • They guide the Normalization process (1NF to BCNF).
Conclusion: Functional dependencies are the semantic constraints of a database. Understanding X → Y is the first and most critical step before attempting to normalize any relation.

Explain First Normal Form (1NF), Second Normal Form (2NF), and Third Normal Form (3NF) with suitable examples.

1. Introduction to Normalization

Normalization is the systematic process of organizing data to minimize redundancy and eliminate undesirable characteristics like Insertion, Update, and Deletion anomalies. It decomposes large tables into smaller, well-structured tables.

2. First Normal Form (1NF)

Rule: A relation is in 1NF if and only if all its attributes are atomic (indivisible). There must be no multi-valued attributes or repeating groups.

Example Problem:

EmpIDNamePhone_Numbers
101Alice1234, 5678

Solution (Convert to 1NF): Create a separate row or table for the multi-valued attribute.

EmpIDNamePhone_Number
101Alice1234
101Alice5678

3. Second Normal Form (2NF)

Rule: A relation is in 2NF if it is in 1NF AND there are no partial dependencies. A partial dependency occurs when a non-prime attribute is functionally dependent on only a part of a composite candidate key.

Example Problem: ENROLL(Student_ID, Course_ID, Course_Name, Grade)

  • Candidate Key: {Student_ID, Course_ID}
  • FD 1: {Student_ID, Course_ID} → Grade (Full dependency)
  • FD 2: Course_ID → Course_Name (Partial dependency: Course_Name depends only on part of the PK).

Solution (Convert to 2NF): Split the table.

  • ENROLL(Student_ID, Course_ID, Grade)
  • COURSE(Course_ID, Course_Name)

4. Third Normal Form (3NF)

Rule: A relation is in 3NF if it is in 2NF AND there are no transitive dependencies. A transitive dependency occurs when a non-prime attribute determines another non-prime attribute (i.e., A → B and B → C, so A → C).

Formal Rule: For every non-trivial FD X → Y, either:
1) X is a superkey, OR
2) Y is a prime attribute (part of a candidate key).

Example Problem: EMPLOYEE(EmpID, Name, DeptID, DeptName)

  • FD 1: EmpID → DeptID
  • FD 2: DeptID → DeptName
  • Therefore, EmpID → DeptName (Transitive dependency).

Solution (Convert to 3NF): Split the table to remove the transitive link.

  • EMPLOYEE(EmpID, Name, DeptID)
  • DEPARTMENT(DeptID, DeptName)
Quick Recall: 1NF eliminates repeating groups. 2NF eliminates partial dependencies. 3NF eliminates transitive dependencies.
Module 4 • Q3

Define Boyce-Codd Normal Form (BCNF). How does it differ from 3NF? Provide an example.

1. Definition of BCNF

Boyce-Codd Normal Form (BCNF) is a stronger, stricter version of 3NF. It was introduced to address specific anomalies that 3NF could not handle, particularly in relations that have overlapping candidate keys.

The BCNF Rule: A relation is in BCNF if and only if for every non-trivial functional dependency X → Y, X is a super key.

Notice that BCNF removes the "escape clause" of 3NF (which allows Y to be a prime attribute). In BCNF, the determinant X MUST be a super key, no exceptions.

2. Difference between 3NF and BCNF

Feature3NFBCNF
Rule for X → Y X is a super key OR Y is a prime attribute. X MUST be a super key.
Strictness Less strict. More strict (Every BCNF is in 3NF, but not vice-versa).
Dependency Preservation 3NF decomposition is always dependency-preserving. BCNF decomposition might sometimes lose dependencies.
Anomaly Handling May still suffer from anomalies if overlapping candidate keys exist. Completely eliminates anomalies caused by functional dependencies.

3. Example of a table in 3NF but NOT in BCNF

Consider a university relation: CLASS(Student, Course, Instructor)

Rules/Semantics:

  • A student takes multiple courses.
  • A course can have multiple instructors.
  • Each instructor teaches exactly ONE course.
  • A student has exactly one instructor for a given course.

Functional Dependencies:

  • FD1: {Student, Course} → Instructor
  • FD2: Instructor → Course

Candidate Keys: {Student, Course} and {Student, Instructor}.

Why is it in 3NF? Look at FD2: Instructor → Course. Is Instructor a superkey? No. Is Course a prime attribute? Yes (it's part of the first candidate key). So it satisfies 3NF.

Why is it NOT in BCNF? Look at FD2 again: Instructor → Course. In BCNF, the determinant (Instructor) MUST be a superkey. It is not. Therefore, it violates BCNF.

4. Conversion to BCNF

To convert it, we decompose based on the violating FD (Instructor → Course):

  • TEACHES(Instructor, Course)
  • ENROLLS(Student, Instructor)
Conclusion: BCNF provides a higher level of normalization by insisting that every determinant must be a candidate key, effectively eliminating anomalies caused by overlapping composite keys.
Module 4 • Q4

Given a relation R(A, B, C, D, E) and FDs {A → B, C → D}, normalize the relation step-by-step to BCNF.

1. Initial Analysis

Relation: R(A, B, C, D, E)

Functional Dependencies (FDs): { A → B, C → D }

2. Step 1: Find the Candidate Key

To find the candidate key, we compute the closure of attributes.

  • We need to derive all attributes {A, B, C, D, E}.
  • A determines B. C determines D. E is not determined by anything, so E must be part of the key.
  • Let's test the closure of {A, C, E}:
    {A, C, E}+ = {A, C, E} (trivial)
    Using A → B: {A, B, C, E}
    Using C → D: {A, B, C, D, E}

Candidate Key: {A, C, E}

3. Step 2: Check for 2NF Violations

A partial dependency occurs when a non-prime attribute depends on part of a candidate key.

  • Candidate Key is {A, C, E}. Prime attributes: A, C, E. Non-prime: B, D.
  • FD A → B: B depends only on A (which is part of the key {A, C, E}). This is a partial dependency.
  • FD C → D: D depends only on C (which is part of the key). This is also a partial dependency.

The relation is not in 2NF. We decompose R to remove these partial dependencies.

Decomposition to 2NF:

  • R1(A, B) with PK: A (Derived from A → B)
  • R2(C, D) with PK: C (Derived from C → D)
  • R3(A, C, E) with PK: {A, C, E} (The remainder relation combining the key attributes)

4. Step 3: Check for 3NF Violations

Are there any transitive dependencies in R1, R2, or R3?

  • R1(A, B): Only FD is A → B. No transitive dependencies.
  • R2(C, D): Only FD is C → D. No transitive dependencies.
  • R3(A, C, E): No FDs defined except trivial ones.

All relations R1, R2, R3 are in 3NF.

5. Step 4: Check for BCNF Violations

In BCNF, for every FD X → Y, X must be a superkey of the relation it is in.

  • In R1(A, B): The FD is A → B. 'A' is the candidate key of R1. Valid BCNF.
  • In R2(C, D): The FD is C → D. 'C' is the candidate key of R2. Valid BCNF.
  • In R3(A, C, E): No non-trivial FDs exist. Valid BCNF.
Final Result: The BCNF decomposition of R(A, B, C, D, E) yields three normalized tables: R1(A, B), R2(C, D), and R3(A, C, E). All partial dependencies have been successfully removed, ensuring a lossless-join decomposition.

Write short notes on: Lossless Join Decomposition and Dependency Preservation.

1. Lossless Join Decomposition

When a large relation R is decomposed into two smaller relations R1 and R2 during normalization, it is absolutely critical that no data is lost and no "fake" data (spurious tuples) is generated when we rejoin them.

Definition: A decomposition of R into R1 and R2 is a Lossless Join Decomposition if the natural join of R1 and R2 yields exactly the original relation R.

R1 ⋈ R2 = R

Condition for Lossless Join:

The decomposition is lossless if the common attributes of R1 and R2 form a candidate key (or superkey) for at least one of the relations. Mathematically, either:

  • (R1 ∩ R2) → R1, or
  • (R1 ∩ R2) → R2

Example: Splitting EMP(ID, Name, DeptID, DeptName) into R1(ID, Name, DeptID) and R2(DeptID, DeptName). The common attribute is DeptID. Since DeptID is the primary key of R2, the condition (R1 ∩ R2) → R2 holds. Therefore, it is lossless.

2. Dependency Preservation

Definition: When a relation R is decomposed, a set of functional dependencies F originally held on R. A decomposition is dependency-preserving if all the functional dependencies in F can still be verified without having to join the decomposed relations.

Why is it important? Enforcing constraints is highly efficient if checking an FD requires looking at only one table. If an FD is split across two tables, the database must perform an expensive JOIN on every INSERT/UPDATE to verify the constraint, which degrades performance.

Example of Loss of Dependency:

Assume R(A, B, C) with FDs A → B and B → C.

  • If we decompose into R1(A, B) and R2(B, C), A → B is checked in R1, and B → C is checked in R2. Dependency is preserved.
  • If we decompose into R1(A, B) and R3(A, C) (due to transitive A → C), we lose the ability to easily verify B → C. Dependency is NOT preserved.
Conclusion: An ideal relational database design must guarantee a Lossless Join to ensure data integrity, and should strongly strive for Dependency Preservation to ensure constraint checking efficiency. Note: 3NF guarantees both, but BCNF guarantees Lossless Join while sometimes sacrificing Dependency Preservation.

INSTRUCTOR_COURSE(Instructor, Course)
STUDENT_INSTRUCTOR(Student, Instructor)

Advantages

  • Removes more redundancy than 3NF.
  • Reduces anomalies significantly.
  • Produces cleaner designs where determinants are keys.

Disadvantages

  • Dependency preservation may not always be possible.
  • Sometimes more joins are required during querying.
Conclusion: BCNF is a stronger form of normalization than 3NF. It requires every determinant to be a super key, thereby minimizing redundancy and dependency-related anomalies.

Module 4 • Q5

Explain lossless join decomposition and dependency preservation with suitable examples.

Introduction

When a relation is decomposed during normalization, two important properties are desirable:

  1. Lossless join decomposition
  2. Dependency preservation

1) Lossless join decomposition

A decomposition of relation R into R1 and R2 is lossless if the natural join of R1 and R2 always reconstructs the original relation exactly, without producing spurious tuples.

R = R1 ⋈ R2 (without spurious tuples)

Condition for binary decomposition

(R1 ∩ R2) → R1 or (R1 ∩ R2) → R2

That is, the common attributes should functionally determine at least one of the decomposed relations.

Example of lossless join

Let R(A, B, C) with FD A → B. Decompose into:

R1(A, B)
R2(A, C)

Common attribute is A. Since A → AB, the decomposition is lossless.

Example of lossy decomposition

R1(A, B)
R2(B, C)

If no FD such as B → A or B → C exists, joining may generate incorrect spurious tuples. Hence it may be lossy.

2) Dependency preservation

A decomposition is dependency preserving if all original functional dependencies can be enforced by checking the decomposed relations individually, without joining them.

If F is the original FD set and F1, F2, ... are projected FDs, then (F1 ∪ F2 ∪ ...)⁺ = F⁺

Example

Suppose R(A, B, C) with FDs A → B and B → C. Decompose into:

R1(A, B)
R2(B, C)

Here, A → B can be checked in R1 and B → C in R2. Hence the decomposition is dependency preserving.

Importance

  • Lossless join ensures correct reconstruction of data.
  • Dependency preservation ensures easy enforcement of constraints.

Trade-off

Sometimes BCNF decomposition is lossless but not dependency preserving. In such cases, designers may prefer 3NF to preserve dependencies while still maintaining a good schema design.

Conclusion: A good decomposition must ideally be both lossless and dependency preserving. Lossless join guarantees correctness of reconstruction, while dependency preservation guarantees correctness of constraint enforcement.

1. Definition of a Transaction

A Transaction is a logical unit of database processing that includes one or more access operations (read, write, insert, delete, update). It is an atomic unit of work that must either be completed entirely or aborted entirely.

2. The State Transition Diagram of a Transaction


Committed Committed Failed Aborted Read/Write ends Commit Errors / Crash Write fail Rollback

3. Example Context

Consider a bank transfer of $500 from Account A (balance $1000) to Account B (balance $2000).

T1:
  Read(A);
  A = A - 500;
  Write(A);
  Read(B);
  B = B + 500;
  Write(B);

4. ACID Properties

To preserve database integrity, the DBMS must ensure four fundamental properties:

A. Atomicity (The "All or Nothing" Rule)

Definition: A transaction is an indivisible unit. Either all its operations are executed, or none are.

Example: If the system crashes after Write(A) but before Write(B), $500 is deducted from A but never added to B. Atomicity ensures the DBMS rolls back the deduction from A, returning it to $1000.

Enforced by: The Transaction Recovery Subsystem.

C. Consistency

Definition: A transaction must transform the database from one valid consistent state to another valid consistent state, adhering to all integrity constraints.

Example: Before T1, total money = 1000 + 2000 = 3000. After T1, A=500, B=2500. Total money = 500 + 2500 = 3000. The consistency of total funds is maintained.

Enforced by: The Application Programmer and DBMS Integrity Constraints.

I. Isolation

Definition: The execution of multiple transactions concurrently should yield the same result as if they were executed sequentially. A transaction should not see the intermediate, uncommitted states of another transaction.

Example: If T2 tries to read A's balance while T1 is mid-execution (A=500, B=2000), T2 sees a temporarily inconsistent database where $500 has vanished. Isolation prevents T2 from reading A until T1 fully commits.

Enforced by: The Concurrency Control Subsystem (via Locking/Timestamps).

D. Durability

Definition: Once a transaction successfully commits, its changes must be permanent and survive any subsequent system crashes (e.g., power failure).

Example: If T1 completes and outputs "Transfer Successful", and the power cord is immediately pulled, the changes (A=500, B=2500) must still be there when the database reboots.

Enforced by: The Recovery Subsystem (via Write-Ahead Logging).

Conclusion: ACID properties are the non-negotiable pillars of transaction processing. Without them, concurrent operations and hardware failures would instantly corrupt the database.
Module 5 • Q2

Define Serializability. Differentiate between Conflict Serializability and View Serializability.

1. Definition of Serializability

Serializability is the property of a concurrent schedule (interleaved execution of multiple transactions) that ensures its final outcome is strictly equivalent to some serial (one-after-another) execution of the same transactions. It guarantees Isolation.

2. Schedule Equivalence Types

To mathematically prove a schedule is serializable, we check for two types of equivalence:

A. Conflict Serializability

Conflict: Two instructions conflict if they belong to different transactions, access the same data item, and at least one is a Write (W) operation. Types of conflicts: Read-Write (R-W), Write-Read (W-R), Write-Write (W-W).

Definition: A schedule S is Conflict Serializable if it can be transformed into a serial schedule by swapping non-conflicting adjacent instructions.

Testing Method (Precedence Graph):

  1. Create a node for each transaction.
  2. Draw a directed edge from Ti to Tj if Ti accesses a data item before Tj, and they conflict.
  3. Rule: If the graph contains a cycle, it is NOT conflict serializable. If it is acyclic, it IS conflict serializable.
B. View Serializability

Definition: View serializability is a slightly more relaxed mathematical condition than conflict serializability. A schedule S is view serializable if it is view equivalent to a serial schedule.

Conditions for View Equivalence (for transactions Ti and Tj):

  1. Initial Read: If Ti reads the initial value of item Q in S1, it must read the initial value of Q in S2.
  2. Update Read: If Ti reads a value of Q produced by Tj in S1, then Ti must read the value of Q produced by Tj in S2.
  3. Final Write: If Ti performs the final write on Q in S1, it must perform the final write on Q in S2.

3. Comparison and Relationship

  • Every Conflict Serializable schedule is automatically View Serializable.
  • Not every View Serializable schedule is Conflict Serializable. (These specifically involve "Blind Writes"—writing a value without reading it first).
  • Testing for Conflict Serializability takes polynomial time (detecting a cycle in a graph). Testing for View Serializability is NP-Complete (computationally intractable for large scales).
Conclusion: Serializability is the benchmark for correct concurrency. Because checking view serializability is computationally explosive, most real-world DBMS engines only test and enforce conflict serializability using locking protocols.
Module 5 • Q3

Explain the Two-Phase Locking (2PL) protocol. What are its variations (Strict and Rigorous)?

1. Introduction to Locking

To guarantee Conflict Serializability automatically, databases use locking. Before a transaction reads/writes a data item, it must acquire a lock on it.

  • Shared Lock (S-lock): Required for reading. Multiple transactions can hold S-locks on the same item simultaneously.
  • Exclusive Lock (X-lock): Required for writing. Only one transaction can hold an X-lock.

2. The Two-Phase Locking (2PL) Protocol

Simply locking and unlocking items is not enough to guarantee serializability; the timing of locks matters. The 2PL protocol dictates that every transaction must be divided into two distinct phases:

  1. Growing Phase: A transaction may obtain new locks (S or X) but may not release any locks.
  2. Lock Point: The moment the transaction acquires its final lock.
  3. Shrinking Phase: A transaction may release locks but may not acquire any new locks.
Mathematical Guarantee: Any schedule produced by transactions following the 2PL protocol is strictly Conflict Serializable.

3. The Problem with Basic 2PL: Cascading Rollbacks

In basic 2PL, a transaction can release an X-lock during its shrinking phase, allowing another transaction to read that uncommitted data. If the first transaction subsequently fails and rolls back, the second transaction read "dirty data" and must also be rolled back. This causes a devastating domino effect called Cascading Rollbacks.

4. Variations of 2PL to Prevent Cascading Rollbacks

A. Strict Two-Phase Locking (Strict 2PL)
  • Rule: A transaction follows basic 2PL, but it holds all its Exclusive (X) locks until the transaction either commits or aborts.
  • Result: No other transaction can read uncommitted modified data. Cascading rollbacks are completely prevented.
B. Rigorous Two-Phase Locking (Rigorous 2PL)
  • Rule: A transaction holds ALL locks (both S and X) until it commits or aborts.
  • Result: Easier to implement than Strict 2PL and guarantees strict serialization order based on commit times, but reduces concurrency (throughput) slightly more than Strict 2PL.

5. Deadlocks in 2PL

2PL does not prevent deadlocks (e.g., T1 holds lock on A waiting for B; T2 holds lock on B waiting for A). Deadlocks must be handled using Deadlock Prevention (Wait-Die, Wound-Wait) or Deadlock Detection (Wait-for graphs).

Conclusion: 2PL is the gold standard protocol for pessimistic concurrency control. While Basic 2PL guarantees serializability, modern databases implement Strict 2PL or Rigorous 2PL to eliminate the destructive problem of cascading rollbacks.

Explain other Concurrency Control Techniques: Timestamp Ordering and Multiversion Concurrency Control (MVCC).

1. Timestamp Ordering Protocol (TO)

Unlike 2PL (which uses locks to delay transactions), the Timestamp Ordering protocol is an optimistic approach that decides the serializability order in advance using system timestamps.

How it works:
  • Every transaction Ti is assigned a unique timestamp TS(Ti) when it starts.
  • Every data item Q maintains two timestamps:
    1) W-timestamp(Q): The largest TS of any transaction that successfully wrote Q.
    2) R-timestamp(Q): The largest TS of any transaction that successfully read Q.
The Core Rules:
  • If Ti wants to Read(Q):
    If TS(Ti) < W-timestamp(Q): This means a "younger" transaction already overwrote the data Ti was supposed to read. Ti is too late. Ti is rejected and rolled back.
    Otherwise: The read is allowed, and R-timestamp(Q) is updated.
  • If Ti wants to Write(Q):
    If TS(Ti) < R-timestamp(Q) OR TS(Ti) < W-timestamp(Q): Ti is trying to write an obsolete value that younger transactions have already interacted with. Ti is rejected and rolled back.
    Otherwise: The write is allowed, and W-timestamp(Q) is updated.

Advantage: Completely deadlock-free because no locks are used. Assures strict serializability.

2. Multiversion Concurrency Control (MVCC)

MVCC is a highly advanced technique used by modern DBMS like PostgreSQL and Oracle. Instead of overwriting data immediately, MVCC maintains multiple historical "versions" of a data item.

How it works:
  • When a transaction writes to item Q, it does not overwrite the old value. It creates a new version Q_new.
  • When a transaction wants to read Q, the DBMS compares the transaction's timestamp against the timestamps of all versions of Q. It serves the specific version of Q that was active precisely when the reading transaction started.
Advantages:
  • Readers never block Writers, and Writers never block Readers.
  • A long-running read query will see a consistent snapshot of the database from the moment it started, unaffected by any new updates happening concurrently.
  • Massively increases throughput and concurrency compared to strict locking.
Conclusion: While 2PL is widely taught, Timestamp Ordering and MVCC offer lock-free alternatives. MVCC, in particular, is the backbone of high-performance modern database systems, trading storage space (for multiple versions) in exchange for exceptional concurrency.

Explain Database Recovery techniques. Write a short note on ARIES and Write-Ahead Logging (WAL).

1. Introduction to Database Recovery

Database Recovery is the process of restoring the database to a correct and consistent state after a system failure (crash, power loss, disk failure). It relies entirely on the system Log File.

2. The System Log

The log is an append-only file kept on stable storage. Every transaction operation is recorded here before it modifies the actual disk. Records look like: [Ti, Write, ItemX, OldValue, NewValue].

3. Write-Ahead Logging (WAL) Protocol

Definition: The WAL protocol is the golden rule of recovery. It states:

  1. Before a transaction's modifications to the database are physically written to disk, the corresponding log records containing the Old and New values MUST be flushed to stable storage.
  2. A transaction cannot be considered "committed" until its COMMIT log record is safely written to disk.

Why? If the system crashes mid-write, the log (which survived) has the OldValue, allowing the system to UNDO the partial write and restore consistency.

4. Basic Recovery Techniques

  • Deferred Update (No-Undo / Redo): Changes are not written to the database until the transaction commits. If it crashes before commit, no UNDO is needed. If it crashes after commit (but before data hit the disk), we REDO from the log.
  • Immediate Update (Undo / Redo): Changes can be written to disk before commit. If it crashes before commit, we must UNDO the changes using OldValue. If it crashes after commit, we REDO using NewValue.

5. The ARIES Recovery Algorithm

ARIES (Algorithm for Recovery and Isolation Exploiting Semantics) is a highly sophisticated, industry-standard recovery algorithm (used in IBM DB2, SQL Server).

Three Phases of ARIES:

  1. Analysis Phase: Scans the log forward from the last checkpoint to identify which transactions were active at the time of the crash and which data pages need recovery.
  2. REDO Phase (Repeating History): Scans the log forward, repeating ALL updates (even those of uncommitted transactions) to bring the database exactly back to the state it was in a millisecond before the crash.
  3. UNDO Phase: Scans the log backward, identifying transactions that were active (did not commit) at crash time, and rolls back their effects one by one using the OldValues.

Key Feature of ARIES: During the UNDO phase, ARIES writes "Compensation Log Records (CLRs)" to the log. This means if the system crashes again while recovering, ARIES will not undo the same action twice.

Conclusion: Recovery guarantees Durability and Atomicity. Using Write-Ahead Logging, algorithms like ARIES can systematically piece together the exact state of the database and roll back incomplete transactions, making databases mathematically resilient to hardware failures.
