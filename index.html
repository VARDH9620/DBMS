Module 1 • Q1

Explain the three-schema architecture of DBMS with a neat diagram. How does it provide logical and physical data independence?

Definition / Introduction

The three-schema architecture, also called the three-level architecture of a DBMS, is a standard framework used to separate user views, overall logical structure, and physical storage details of a database. Its main purpose is to achieve data abstraction and data independence.

Neat diagram: Three-schema architecture

Explanation of the three levels

  1. External level (View level): It contains multiple user views. Each user or application sees only the required part of the database.
  2. Conceptual level (Logical level): It describes the complete logical structure of the database such as entities, attributes, relationships, and constraints.
  3. Internal level (Physical level): It specifies how data is actually stored on disk, for example files, indexes, hashing, compression, and storage blocks.

Mappings in the architecture

  • External–conceptual mapping: Connects each user view to the conceptual schema.
  • Conceptual–internal mapping: Connects logical structure to physical storage.

How it provides data independence

Data independence means that a change at one schema level should not force changes at the next higher level.

  1. Physical data independence: Changes in the internal schema do not affect the conceptual schema.
    • Example: changing file organization from sequential to B+ tree indexing.
    • Example: moving data to another disk or adding indexes for faster retrieval.
  2. Logical data independence: Changes in the conceptual schema do not affect external schemas or application programs.
    • Example: adding a new attribute Email to STUDENT should not disturb an old payroll or results application that does not use it.
    • Example: splitting one large relation into two related relations while preserving user views.

Advantages

  • Provides high level of abstraction.
  • Improves security because users see only required views.
  • Reduces application maintenance.
  • Makes database design modular and flexible.

Disadvantages / Limitations

  • Implementation complexity is high.
  • Complete logical data independence is difficult to achieve in practice.
  • Mapping overhead may add design complexity.

Applications

This architecture is used in banking systems, ERP applications, university information systems, hospital management systems, and enterprise databases where multiple users require different views of the same data.

Conclusion: Thus, the three-schema architecture is a fundamental DBMS design model that separates view, logical, and physical levels. By isolating these levels through mappings, it achieves logical and physical data independence and makes databases easier to maintain and scale.
Module 1 • Q2

Define data independence. Explain logical data independence and physical data independence with suitable examples.

Definition

Data independence is the ability to change the schema at one level of a database system without requiring changes at the next higher level. It is one of the major objectives of DBMS design.

Need for data independence

  • Database structure changes over time.
  • Storage methods improve due to optimization.
  • Applications should continue working with minimum modification.

Types of data independence

1) Physical data independence

Physical data independence is the immunity of the conceptual schema from changes in the internal schema.

In simple words: changes in storage should not affect the logical structure of the database.

Examples

  1. Changing the file organization from heap file to indexed file.
  2. Adding B-tree or hash index on a relation.
  3. Changing block size or storage location.
  4. Using compression or partitioning for performance.

Observation: user queries such as SELECT * FROM STUDENT remain unchanged.

2) Logical data independence

Logical data independence is the immunity of external schemas and application programs from changes in the conceptual schema.

In simple words: changes in the logical structure should not affect user views.

Examples

  1. Adding a new attribute Phone_No to STUDENT.
  2. Splitting relation EMPLOYEE(EmpID, Name, DeptName) into EMPLOYEE(EmpID, Name, DeptID) and DEPARTMENT(DeptID, DeptName).
  3. Introducing a new relation while keeping existing views unchanged.

Comparison between logical and physical data independence

BasisPhysical Data IndependenceLogical Data Independence
Level affectedInternal levelConceptual level
Higher level protectedConceptual schemaExternal schema / programs
Ease of achievementRelatively easierRelatively difficult
ExampleAdd indexAdd attribute or split relation

Advantages

  • Improves portability and longevity of applications.
  • Reduces maintenance cost.
  • Supports database tuning without disturbing users.
  • Enables gradual schema evolution.

Limitations

  • Not absolute in real systems.
  • Major logical redesign may still force application changes.
  • Legacy systems often have weaker logical data independence.
Conclusion: Hence, data independence is the key feature that distinguishes DBMS from traditional file systems. Physical data independence hides storage changes, whereas logical data independence hides changes in the logical structure from application programs.
Module 1 • Q3

What is an Entity-Relationship (ER) model? Draw a neat ER diagram for a Library Management System and explain its components.

Definition / Introduction

The Entity-Relationship (ER) model is a high-level conceptual data model used for database design. It represents real-world objects as entities, their properties as attributes, and associations among them as relationships.

Main components of ER model

  1. Entity: A distinguishable real-world object such as Book, Member, or Librarian.
  2. Attribute: A property of an entity such as Book_ID, Title, Name, Phone.
  3. Relationship: Association among entities such as a Member borrows a Book_Copy.
  4. Key attribute: An attribute that uniquely identifies entity instances.
  5. Constraints: Cardinality and participation restrictions.
ER diagram for Library Management System

Explanation of the diagram

  1. BOOK stores bibliographic details of a title. ISBN is the key attribute.
  2. COPY represents physical copies of a book. One book can have many copies.
  3. MEMBER stores student/staff information who can borrow books.
  4. LOAN is an associative entity that records borrowing transactions including issue date, due date, return date, and fine.

Cardinality and participation

  • BOOK to COPY = 1:N.
  • MEMBER to LOAN = 1:N.
  • COPY to LOAN = 1:N over time, but typically one active loan at a given instant.

Advantages of ER model

  • Easy to understand and communicate with users.
  • Captures real-world semantics effectively.
  • Acts as a bridge between requirement analysis and relational design.

Applications

ER modeling is used in library systems, banking systems, hospital databases, inventory management, and online commerce applications.

Conclusion: Therefore, the ER model is a powerful conceptual tool for representing entities, attributes, relationships, and constraints. The Library Management ER diagram clearly shows how books, copies, members, and loans are interconnected in a practical database design.
Module 1 • Q4

Explain different types of attributes, relationships, and constraints used in ER modeling with examples.

Introduction

ER modeling uses three major building blocks: attributes, relationships, and constraints. Proper understanding of these elements is necessary for correct conceptual database design.

1) Types of attributes

  1. Simple (atomic) attribute: cannot be subdivided. Example: Age, Salary.
  2. Composite attribute: can be broken into components. Example: Address = House_No, Street, City, PIN.
  3. Single-valued attribute: has only one value for each entity. Example: Date_of_Birth.
  4. Multi-valued attribute: can have multiple values. Example: Phone_Numbers, Skills.
  5. Derived attribute: obtained from another attribute. Example: Age derived from Date_of_Birth.
  6. Key attribute: uniquely identifies an entity. Example: USN in STUDENT.
  7. Null-valued attribute: value may be unknown or not applicable. Example: Middle_Name.

2) Types of relationships

  1. Unary (recursive) relationship: relationship among entities of the same set. Example: EMPLOYEE supervises EMPLOYEE.
  2. Binary relationship: between two entity sets. Example: STUDENT enrolls in COURSE.
  3. Ternary relationship: among three entity sets. Example: SUPPLIER supplies PART to PROJECT.
  4. Relationship with attributes: Example: WORKS_ON has attribute Hours.

Cardinality ratios of relationships

  • 1:1 – One department has one manager.
  • 1:N – One department has many employees.
  • M:N – One student enrolls in many courses and one course has many students.

3) Constraints in ER modeling

  1. Cardinality constraint: specifies number of entities participating in a relationship.
    • Example: One customer can place many orders.
  2. Participation constraint: specifies whether participation is total or partial.
    • Total participation: every EMPLOYEE must belong to some DEPARTMENT.
    • Partial participation: only some EMPLOYEES manage departments.
  3. Key constraint: ensures uniqueness through key attributes.
  4. Existence dependency: weak entity depends on owner entity. Example: DEPENDENT depends on EMPLOYEE.

Advantages of proper constraint specification

  • Improves semantic correctness of design.
  • Reduces ambiguity in requirements.
  • Helps during relational schema conversion.

Applications

These concepts are essential in designing academic, banking, airline, manufacturing, and healthcare databases.

Conclusion: In ER modeling, attributes describe properties, relationships describe associations, and constraints specify the rules of participation and cardinality. Together, they produce a precise conceptual representation of the application domain.

Introduction

Before the evolution of DBMS, organizations stored data in separate application-oriented files. This approach is known as the traditional file processing system. A DBMS is a software system that stores, organizes, controls, and retrieves data in an integrated manner.

Problems in file processing system

  • Data redundancy and inconsistency
  • Difficulty in accessing data
  • Data isolation across different files and formats
  • Integrity problems

  • Weak security

  1. Reduced redundancy: same data need not be stored in many files.

Module 2 • Q1

Explain the fundamental operations of Relational Algebra with suitable examples.

Definition / Introduction

Relational Algebra is a procedural query language used to specify what operations are to be performed on relations. It forms the theoretical foundation of relational DBMS query processing.

Fundamental operations

  1. Selection σ
  2. Projection π
  3. Union ∪
  4. Set Difference −
  5. Cartesian Product ×
  6. Rename ρ

Formal notation and examples

Assume relation STUDENT(USN, Name, Dept, Sem).

Selection: σp(R) = { t | t ∈ R and predicate p(t) is true } Example: σDept='CSE'(STUDENT)

Meaning: selects only those tuples where department is CSE.

Projection: πA1, A2, ..., An(R) Example: πUSN, Name(STUDENT)

Meaning: retrieves only specified columns and removes duplicates.

Union: R ∪ S Condition: R and S must be union-compatible.

Example: πUSN(PLACED_STUDENTS) ∪ πUSN(HIGHER_STUDIES)

Set Difference: R − S Meaning: tuples in R but not in S

Example: students registered in DBMS but not in OS.

Cartesian Product: R × S

Meaning: combines every tuple of R with every tuple of S. It is usually followed by selection to realize joins.

Rename: ρX(R) or ρX(A1, A2, ..., An)(R)

Meaning: changes relation name or attribute names.

Derived operations

  • Intersection: R ∩ S = R − (R − S)
  • Join: R ⋈ S
  • Natural join, theta join, division are also widely used derived operations.

Illustrative example of join

Let STUDENT(USN, Name, DeptID) and DEPT(DeptID, DeptName).

STUDENT ⋈STUDENT.DeptID = DEPT.DeptID DEPT

This gives student details along with department name.

Advantages

  • Provides mathematical basis for query processing.
  • Useful for query optimization.
  • Independent of implementation details.

Applications

Relational algebra is used internally by query optimizers and is essential for understanding SQL translation, joins, views, and database theory.

Conclusion: Therefore, relational algebra provides a set of fundamental and derived operations that manipulate relations systematically. It is the formal backbone of relational query processing.
Module 2 • Q2

Define Super Key, Candidate Key, Primary Key, Alternate Key, and Foreign Key. Explain each with suitable examples.

Introduction

Keys are attributes or sets of attributes used to uniquely identify tuples in a relation and to establish relationships among relations.

Consider relation STUDENT(USN, Aadhaar, Email, Name, DeptID), where USN, Aadhaar, and Email are unique.

1) Super Key

A super key is any set of one or more attributes that uniquely identifies a tuple.

Examples: {USN}, {Aadhaar}, {Email}, {USN, Name}.

2) Candidate Key

A candidate key is a minimal super key, that is, no proper subset of it can uniquely identify tuples.

Examples: {USN}, {Aadhaar}, {Email}.

3) Primary Key

A primary key is the candidate key selected by the designer to identify tuples uniquely.

Example: choosing USN as primary key.

4) Alternate Key

An alternate key is any candidate key that is not chosen as the primary key.

Example: if USN is primary key, then Aadhaar and Email are alternate keys.

5) Foreign Key

A foreign key is an attribute in one relation that refers to the primary key of another relation.

Consider DEPARTMENT(DeptID, DeptName). In STUDENT(..., DeptID), the attribute DeptID is a foreign key referencing DEPARTMENT(DeptID).

Tabular comparison

Key TypePurposeExample
Super KeyUniquely identifies tuple{USN, Name}
Candidate KeyMinimal super key{USN}
Primary KeyMain selected identifierUSN
Alternate KeyUnused candidate keyEmail
Foreign KeyReferences another relationDeptID

Importance of keys

  • Ensure uniqueness.
  • Prevent duplicate tuples.
  • Support referential integrity.
  • Help in indexing and efficient retrieval.
Conclusion: Keys are the backbone of relational database design. They uniquely identify tuples, maintain integrity, and create meaningful links among relations.
Module 2 • Q3

Explain the relational data model and its integrity constraints with examples.

Definition

The relational data model represents data as a collection of relations (tables). Each relation consists of rows called tuples and columns called attributes.

Basic terms

  • Relation: table
  • Tuple: row
  • Attribute: column
  • Domain: set of valid values for an attribute
  • Schema: structure of a relation
  • Instance: current contents of a relation

Example

STUDENT(USN, Name, DeptID, Sem)
DEPARTMENT(DeptID, DeptName)
ENROLL(USN, CID, Marks)

Integrity constraints in relational model

  1. Domain constraint
    • Each attribute must take values only from its specified domain.
    • Example: Sem must be between 1 and 8.
  2. Key constraint
    • No two tuples can have the same primary key value.
    • Example: no two students can have the same USN.
  3. Entity integrity constraint
    • Primary key cannot be null.
    • Example: USN in STUDENT must always have a value.
  4. Referential integrity constraint
    • A foreign key value must either match an existing primary key value in the referenced relation or be null, if allowed.
    • Example: DeptID in STUDENT must exist in DEPARTMENT.
  5. General / semantic constraints
    • User-defined business rules.
    • Example: Marks must be between 0 and 100.

Advantages of relational model

  • Simple tabular representation.
  • Strong mathematical foundation.
  • Easy querying using SQL.
  • Reduced redundancy through normalization.

Disadvantages

  • Complex joins may reduce performance.
  • Hierarchical or graph-like data may be less natural to model.
Conclusion: The relational data model represents data in tables and maintains correctness through integrity constraints such as domain, key, entity integrity, and referential integrity constraints.
Module 2 • Q4

Explain the process of mapping an ER model into a relational schema with suitable examples.

Introduction

ER-to-relational mapping is the process of converting a conceptual ER design into implementable relational tables. The purpose is to transform entities, relationships, and attributes into relations, primary keys, and foreign keys.

Mapping flow from ER model to relational schema

Step-by-step mapping rules

  1. Map regular (strong) entity type
    • Create one relation for each entity.
    • Include all simple attributes.
    • Choose the key attribute as primary key.
    • Example: STUDENT(USN, Name, Sem)
  2. Map weak entity type
    • Create a relation for the weak entity.
    • Include its partial key and the owner’s primary key as foreign key.
    • The combined key becomes primary key.
    • Example: DEPENDENT(EmpID, DepName, Age), PK = (EmpID, DepName)
  3. Map binary 1:1 relationship
    • Add foreign key of one relation to the other, preferably on the side with total participation.
    • Relationship attributes may also be stored there.
  4. Map binary 1:N relationship
    • Add primary key of the 1-side as a foreign key to the N-side relation.
    • Example: DEPARTMENT(DeptID) and EMPLOYEE(..., DeptID)
  5. Map binary M:N relationship
    • Create a new relation.
    • Include primary keys of participating relations as foreign keys.
    • Include relationship attributes.
    • Example: ENROLL(USN, CID, Marks)
  6. Map multivalued attribute
    • Create a separate relation consisting of key of owner entity + multivalued attribute.
    • Example: STUDENT_PHONE(USN, PhoneNo)
  7. Map n-ary relationship
    • Create a new relation containing keys of all participating entities plus relationship attributes.

Illustrative example

Suppose the ER model contains STUDENT, COURSE, and relationship ENROLLS with attribute Grade.

STUDENT(USN, Name, Sem)
COURSE(CID, Title, Credits)
ENROLL(USN, CID, Grade)

Here, ENROLL is created because the relationship is M:N.

Advantages

  • Systematic conversion of conceptual design into physical implementation.
  • Preserves semantics through keys and constraints.
  • Supports normalization and integrity checking.
Conclusion: ER-to-relational mapping converts strong entities, weak entities, and relationships into tables with primary keys and foreign keys. It is an essential step in logical database design.

Module 3: SQL

Module 3 • Q1

Explain the SQL language categories DDL, DML, DCL, and TCL with suitable commands and examples.

Introduction

SQL (Structured Query Language) is the standard language used for defining, manipulating, controlling, and managing data in relational databases. SQL commands are broadly classified into DDL, DML, DCL, and TCL.

1) DDL – Data Definition Language

DDL commands are used to define or modify the structure of database objects such as tables, views, schemas, and indexes.

Common commands: CREATE, ALTER, DROP, TRUNCATE, RENAME

CREATE TABLE Student(
  USN VARCHAR(10) PRIMARY KEY,
  Name VARCHAR(30),
  DeptID INT,
  Sem INT
);

ALTER TABLE Student ADD Email VARCHAR(40);
DROP TABLE Student;

2) DML – Data Manipulation Language

DML commands are used to insert, modify, delete, and retrieve data from tables.

Common commands: INSERT, UPDATE, DELETE, SELECT

INSERT INTO Student VALUES('1RV23CS001', 'Asha', 10, 3);
UPDATE Student SET Sem = 4 WHERE USN = '1RV23CS001';
DELETE FROM Student WHERE USN = '1RV23CS001';
SELECT * FROM Student;

3) DCL – Data Control Language

DCL commands control access privileges on database objects.

Common commands: GRANT, REVOKE

GRANT SELECT, INSERT ON Student TO faculty_user;
REVOKE INSERT ON Student FROM faculty_user;

4) TCL – Transaction Control Language

TCL commands manage transactions and ensure consistency of operations.

Common commands: COMMIT, ROLLBACK, SAVEPOINT

UPDATE Account SET Balance = Balance - 500 WHERE AccNo = 101;
SAVEPOINT s1;
UPDATE Account SET Balance = Balance + 500 WHERE AccNo = 202;
COMMIT;
-- or ROLLBACK TO s1;

Comparison table

CategoryPurposeExamples
DDLDefines structureCREATE, ALTER, DROP
DMLManipulates dataSELECT, INSERT, UPDATE, DELETE
DCLControls privilegesGRANT, REVOKE
TCLControls transactionsCOMMIT, ROLLBACK, SAVEPOINT

Applications

Together these categories cover schema creation, data processing, security management, and transaction management in all practical relational database systems.

Conclusion: SQL is not a single-purpose language. DDL defines structures, DML manipulates data, DCL manages permissions, and TCL ensures reliable transaction execution.
Module 3 • Q2

Write SQL queries for the following operations using a Student database: Joins, Aggregate Functions, Nested Queries, GROUP BY and HAVING clauses.

Assumed schema: Student(USN, Name, DeptID, Sem), Department(DeptID, DeptName), Course(CID, Title, Credits), Enroll(USN, CID, Marks).

1) Joins

Question: Display student name, department name, course title, and marks.

SELECT s.USN, s.Name, d.DeptName, c.Title, e.Marks
FROM Student s
JOIN Department d ON s.DeptID = d.DeptID
JOIN Enroll e ON s.USN = e.USN
JOIN Course c ON e.CID = c.CID;

Explanation: The query joins four related tables to display integrated information from multiple relations.

2) Aggregate functions

SELECT AVG(Marks) AS AvgMarks,
       MAX(Marks) AS Highest,
       MIN(Marks) AS Lowest,
       COUNT(*)   AS TotalEnrollments
FROM Enroll;

This query calculates average, maximum, minimum, and total number of enrollment records.

3) Nested query

Question: Find names of students who scored above the average marks.

SELECT s.Name, e.Marks
FROM Student s
JOIN Enroll e ON s.USN = e.USN
WHERE e.Marks > (SELECT AVG(Marks) FROM Enroll);

The inner query finds average marks; the outer query selects students whose marks exceed that average.

4) GROUP BY

Question: Find department-wise average marks.

SELECT d.DeptName, AVG(e.Marks) AS AvgMarks
FROM Student s
JOIN Department d ON s.DeptID = d.DeptID
JOIN Enroll e ON s.USN = e.USN
GROUP BY d.DeptName;

5) HAVING clause

Question: Display only those departments whose average marks are above 75.

SELECT d.DeptName, AVG(e.Marks) AS AvgMarks
FROM Student s
JOIN Department d ON s.DeptID = d.DeptID
JOIN Enroll e ON s.USN = e.USN
GROUP BY d.DeptName
HAVING AVG(e.Marks) > 75;

Step-by-step note for exam

  1. Use JOIN when data is distributed across multiple tables.
  2. Use aggregate functions such as COUNT, SUM, AVG, MAX, MIN for summary data.
  3. Use nested queries when one query depends on the result of another.
  4. Use GROUP BY to form groups.
  5. Use HAVING to filter groups, whereas WHERE filters rows before grouping.
Conclusion: SQL provides powerful constructs such as joins, aggregates, subqueries, grouping, and HAVING clauses to answer simple as well as complex data retrieval requirements efficiently.
Module 3 • Q3

What are Views in SQL? Explain the creation, modification, and deletion of views with suitable examples.

Definition

A view is a virtual table whose contents are derived from a SQL query. It does not necessarily store data physically; instead, it presents data from one or more base tables in a customized form.

Need for views

  • To simplify complex queries.
  • To provide security by exposing limited columns/rows.
  • To hide implementation complexity.
  • To present logical subsets of data to different users.

Syntax for creation of a view

CREATE VIEW view_name AS
SELECT column1, column2, ...
FROM table_name
WHERE condition;

Example: creation

CREATE VIEW CSE_Students AS
SELECT s.USN, s.Name, d.DeptName
FROM Student s
JOIN Department d ON s.DeptID = d.DeptID
WHERE d.DeptName = 'CSE';

This view shows only CSE students.

Modification of a view

In many DBMS products, modification is done using CREATE OR REPLACE VIEW or ALTER VIEW.

CREATE OR REPLACE VIEW CSE_Students AS
SELECT s.USN, s.Name, s.Sem, d.DeptName
FROM Student s
JOIN Department d ON s.DeptID = d.DeptID
WHERE d.DeptName = 'CSE';

Now the view additionally includes semester.

Deletion of a view

DROP VIEW CSE_Students;

Types of views

  1. Simple view: based on a single table, often updatable.
  2. Complex view: based on multiple tables, joins, aggregates, or grouping; generally read-only unless specific mechanisms are used.

Advantages

  • Improves security.
  • Reduces query complexity.
  • Provides logical data independence.
  • Helps reuse common queries.

Disadvantages

  • Complex views may not be updatable.
  • Nested views can affect performance and readability.

Applications

Views are widely used in reports, dashboards, restricted user access, academic portals, ERP systems, and data abstraction layers.

Conclusion: A view is a virtual table created from a query. It is useful for simplification, security, and abstraction, and can be created, modified, and deleted using standard SQL commands.
Module 3 • Q4

What are Triggers? Explain the syntax, working, and applications of triggers with suitable examples.

Definition

A trigger is a database object or procedural program that is automatically executed by the DBMS in response to specified events such as INSERT, UPDATE, or DELETE on a table or view.

General syntax

CREATE TRIGGER trigger_name
BEFORE | AFTER | INSTEAD OF
INSERT OR UPDATE OR DELETE
ON table_name
FOR EACH ROW
BEGIN
   -- trigger body
END;

Classification of triggers

  • BEFORE trigger: executed before the event.
  • AFTER trigger: executed after the event.
  • INSTEAD OF trigger: often used on views.
  • Row-level trigger: fires once for each affected row.
  • Statement-level trigger: fires once for the whole SQL statement.

Working of a trigger

  1. An event such as insert/update/delete occurs.
  2. The DBMS checks whether a matching trigger exists.
  3. The trigger body executes automatically.
  4. It may validate data, modify values, log changes, or prevent invalid operations.

Example 1: validation trigger

CREATE TRIGGER Check_Marks
BEFORE INSERT ON Enroll
FOR EACH ROW
BEGIN
  IF NEW.Marks < 0 OR NEW.Marks > 100 THEN
    SIGNAL SQLSTATE '45000'
    SET MESSAGE_TEXT = 'Marks must be between 0 and 100';
  END IF;
END;

This trigger prevents invalid marks from being inserted.

Example 2: audit trigger

CREATE TRIGGER Audit_Student_Update
AFTER UPDATE ON Student
FOR EACH ROW
BEGIN
  INSERT INTO Student_Audit(USN, OldName, NewName, ChangedOn)
  VALUES(OLD.USN, OLD.Name, NEW.Name, CURRENT_TIMESTAMP);
END;

This trigger stores an audit trail whenever student data changes.

Advantages

  • Automatic enforcement of business rules.
  • Useful for auditing and logging.
  • Maintains derived values and consistency.
  • Reduces repetitive application-side checks.

Disadvantages

  • May become difficult to debug.
  • Hidden logic can reduce maintainability.
  • Excessive triggers may affect performance.

Applications

  • Audit logging
  • Security checks
  • Automatic timestamp updates
  • Validation of salary, marks, stock quantity, etc.
  • Implementing update behavior on complex views
Conclusion: Triggers are event-driven database procedures that automatically execute in response to data changes. They are extremely useful for validation, auditing, and automated rule enforcement.

Consider a student database. Write SQL queries to perform the following operations and explain the output: highest scorer, department-wise student count, students enrolled in multiple courses, second highest marks.

Module 4: Functional Dependencies & Normalization

Module 4 • Q1

Define Functional Dependency. Explain trivial, non-trivial, and fully functional dependencies with suitable examples.

Definition

A functional dependency (FD) is a relationship between attributes in a relation. For attributes X and Y of a relation R, the dependency X → Y means that whenever two tuples have the same value of X, they must also have the same value of Y.

If t1[X] = t2[X], then t1[Y] = t2[Y] Therefore, X → Y

Example relation

Consider STUDENT(USN, Name, DeptID, DeptName).

  • USN → Name, DeptID
  • DeptID → DeptName

1) Trivial functional dependency

A dependency X → Y is trivial if Y ⊆ X.

Examples: {USN, Name} → Name, {A, B} → A

2) Non-trivial functional dependency

A dependency X → Y is non-trivial if Y is not a subset of X.

Examples: USN → Name, DeptID → DeptName

3) Fully functional dependency

Y is said to be fully functionally dependent on X if X → Y holds, but no proper subset of X determines Y.

Consider RESULT(USN, CID, Marks). Here:

  • (USN, CID) → Marks
  • Neither USN → Marks nor CID → Marks
  • Hence, Marks is fully functionally dependent on (USN, CID).

Importance of FD

  • Basis for identifying keys
  • Essential for normalization
  • Helps reduce redundancy and anomalies
Conclusion: Functional dependency expresses association among attributes. Trivial and non-trivial dependencies classify FDs, while fully functional dependency is especially important in removing partial dependency during normalization.
Module 4 • Q2

Explain First Normal Form (1NF), Second Normal Form (2NF), and Third Normal Form (3NF) with suitable examples.

Introduction

Normalization is the process of organizing data in relations to minimize redundancy and eliminate insertion, deletion, and update anomalies.

Progression of normalization

Example unnormalized relation

Consider: STUDENT_COURSE(USN, Name, DeptName, CID, CTitle, Instructor, Marks)

Assume key = (USN, CID).

1) First Normal Form (1NF)

A relation is in 1NF if all attributes contain atomic values and there are no repeating groups or multivalued attributes.

  • Each field must contain only one value.
  • Example: instead of storing Phone = {9876, 9123}, store them separately or in another relation.

2) Second Normal Form (2NF)

A relation is in 2NF if it is in 1NF and every non-prime attribute is fully functionally dependent on the whole candidate key.

Partial dependencies in the example:

  • USN → Name, DeptName
  • CID → CTitle, Instructor

These depend on only part of the composite key, so the relation is not in 2NF.

Decomposition:

STUDENT(USN, Name, DeptName)
COURSE(CID, CTitle, Instructor)
ENROLL(USN, CID, Marks)

3) Third Normal Form (3NF)

A relation is in 3NF if it is in 2NF and has no transitive dependency of non-key attributes on a key.

Suppose in STUDENT relation, USN → DeptID and DeptID → DeptName. Then USN → DeptName is a transitive dependency.

3NF decomposition:

STUDENT(USN, Name, DeptID)
DEPARTMENT(DeptID, DeptName)
COURSE(CID, CTitle, Instructor)
ENROLL(USN, CID, Marks)

Advantages of normalization

  • Reduces redundancy
  • Eliminates anomalies
  • Improves consistency
  • Simplifies maintenance
Conclusion: 1NF removes repeating groups, 2NF removes partial dependencies, and 3NF removes transitive dependencies. Together, they significantly improve database design quality.
Module 4 • Q3

What is Boyce-Codd Normal Form (BCNF)? Explain the conditions for BCNF with an example.

Definition

A relation is in Boyce-Codd Normal Form (BCNF) if for every non-trivial functional dependency X → Y in the relation, X is a super key.

A relation R is in BCNF if for every non-trivial FD X → Y, X is a super key of R.

Conditions for BCNF

  1. The dependency must be non-trivial.
  2. The determinant on the left-hand side must be a super key.

Why BCNF is stronger than 3NF

3NF allows some dependencies where the right-hand side attribute is prime. BCNF does not allow that exception. Hence BCNF is stricter and reduces redundancy further.

Example

Consider relation TEACHING(Student, Course, Instructor) with FDs:

  • (Student, Course) → Instructor
  • Instructor → Course

Suppose one instructor teaches only one course. Then Instructor → Course holds.

Here, Instructor is not a super key, so the relation violates BCNF.

BCNF decomposition

INSTRUCTOR_COURSE(Instructor, Course)
STUDENT_INSTRUCTOR(Student, Instructor)

Advantages

  • Removes more redundancy than 3NF.
  • Reduces anomalies significantly.
  • Produces cleaner designs where determinants are keys.

Disadvantages

  • Dependency preservation may not always be possible.
  • Sometimes more joins are required during querying.
Conclusion: BCNF is a stronger form of normalization than 3NF. It requires every determinant to be a super key, thereby minimizing redundancy and dependency-related anomalies.

Normalize the given relation step-by-step up to Third Normal Form (3NF) or BCNF, showing all intermediate steps.

Assumption: Since the PDF does not include a specific relation, the following standard example is used to demonstrate the complete method.

Given relation

ENROLLMENT(SID, SName, DeptID, DeptName, CID, CName, InstructorID, InstructorName, Grade)

Functional dependencies:

  • SID → SName, DeptID
  • DeptID → DeptName
  • CID → CName, InstructorID
  • InstructorID → InstructorName
  • (SID, CID) → Grade

Candidate key: (SID, CID)

Step 1: Check for 1NF

All attributes are atomic. Therefore, the relation is in 1NF.

Step 2: Convert 1NF to 2NF

Since the candidate key is composite (SID, CID), we check for partial dependencies.

  • SID → SName, DeptID is partial dependency.
  • CID → CName, InstructorID is partial dependency.

Hence, decompose into:

STUDENT(SID, SName, DeptID)
COURSE(CID, CName, InstructorID)
ENROLLMENT(SID, CID, Grade, DeptName, InstructorName)

But to avoid mixing transitively dependent attributes, it is better to isolate them properly in the next step.

Step 3: Convert to proper 2NF structure

STUDENT(SID, SName, DeptID)
COURSE(CID, CName, InstructorID)
ENROLLMENT(SID, CID, Grade)

Now all non-key attributes in each relation depend on the whole key. So these are in 2NF.

Step 4: Convert 2NF to 3NF

Check for transitive dependencies:

  • In STUDENT: SID → DeptID and DeptID → DeptName

STUDENT(SID, SName, DeptID)

ENROLLMENT(SID, CID, Grade)

  • In STUDENT, determinant SID is a key.
  • In DEPARTMENT, determinant DeptID is a key.
  • In COURSE, determinant CID is a key.
  • In INSTRUCTOR, determinant InstructorID is a key.
  • In ENROLLMENT, determinant (SID, CID) is a key.

Therefore, all resulting relations are also in BCNF.

Final normalized set

STUDENT(SID, SName, DeptID)
DEPARTMENT(DeptID, DeptName)
COURSE(CID, CName, InstructorID)
INSTRUCTOR(InstructorID, InstructorName)
ENROLLMENT(SID, CID, Grade)

Benefits obtained

  • No update anomaly when department name changes.
  • No insertion anomaly for adding a new instructor or department.
  • No deletion anomaly causing accidental loss of department/instructor information.

Module 4 • Q5

Explain lossless join decomposition and dependency preservation with suitable examples.

Introduction

When a relation is decomposed during normalization, two important properties are desirable:

  1. Lossless join decomposition
  2. Dependency preservation

1) Lossless join decomposition

A decomposition of relation R into R1 and R2 is lossless if the natural join of R1 and R2 always reconstructs the original relation exactly, without producing spurious tuples.

R = R1 ⋈ R2 (without spurious tuples)

Condition for binary decomposition

(R1 ∩ R2) → R1 or (R1 ∩ R2) → R2

That is, the common attributes should functionally determine at least one of the decomposed relations.

Example of lossless join

Let R(A, B, C) with FD A → B. Decompose into:

R1(A, B)
R2(A, C)

Common attribute is A. Since A → AB, the decomposition is lossless.

Example of lossy decomposition

R1(A, B)
R2(B, C)

If no FD such as B → A or B → C exists, joining may generate incorrect spurious tuples. Hence it may be lossy.

2) Dependency preservation

A decomposition is dependency preserving if all original functional dependencies can be enforced by checking the decomposed relations individually, without joining them.

If F is the original FD set and F1, F2, ... are projected FDs, then (F1 ∪ F2 ∪ ...)⁺ = F⁺

Example

Suppose R(A, B, C) with FDs A → B and B → C. Decompose into:

R1(A, B)
R2(B, C)

Here, A → B can be checked in R1 and B → C in R2. Hence the decomposition is dependency preserving.

Importance

  • Lossless join ensures correct reconstruction of data.
  • Dependency preservation ensures easy enforcement of constraints.

Trade-off

Sometimes BCNF decomposition is lossless but not dependency preserving. In such cases, designers may prefer 3NF to preserve dependencies while still maintaining a good schema design.

Conclusion: A good decomposition must ideally be both lossless and dependency preserving. Lossless join guarantees correctness of reconstruction, while dependency preservation guarantees correctness of constraint enforcement.

Module 5: Transactions, Concurrency Control & Recovery

Module 5 • Q1

What is a transaction? Explain the ACID properties of transactions with suitable examples.

Definition

A transaction is a logical unit of work that consists of one or more database operations and must be executed completely or not at all. A transaction takes the database from one consistent state to another consistent state.

Example: Bank transfer transaction

Read balance(A)
A = A - 500
Write balance(A)
Read balance(B)
B = B + 500
Write balance(B)
COMMIT

Either all these operations must occur, or none must occur.

ACID properties

  1. Atomicity
    • All operations of a transaction are performed completely, or none are performed.
    • If a failure occurs after debiting A but before crediting B, the system must roll back the debit.
  2. Consistency
    • A transaction must preserve database rules and constraints.
    • Example: total money before and after transfer should remain the same.
  3. Isolation
    • Concurrent transactions should not interfere in a way that produces incorrect results.
    • Each transaction should appear as if it is executing alone.
  4. Durability
    • Once a transaction commits, its effects must persist even if a crash occurs immediately afterward.

Transaction states

  • Active
  • Partially committed
  • Committed
  • Failed
  • Aborted

Advantages of transactions

  • Ensures reliability of updates.
  • Prevents inconsistent data.
  • Supports safe multi-user operations.
Conclusion: A transaction is a reliable unit of database work. Its correctness is ensured by the ACID properties: Atomicity, Consistency, Isolation, and Durability.
Module 5 • Q2

Define serializability. Explain conflict serializability and view serializability with suitable examples.

Definition

Serializability is the property that a concurrent schedule should produce the same effect as some serial execution of the same transactions. It is the fundamental correctness criterion for concurrent transactions.

1) Conflict serializability

A schedule is conflict serializable if it can be transformed into a serial schedule by repeatedly swapping non-conflicting operations.

Two operations conflict if:

  • They belong to different transactions,
  • They access the same data item, and
  • At least one of them is a write operation.

Example of conflict-serializable schedule

S1: R1(X)  W1(X)  R2(X)  W2(X)

Conflicting order forces T1 → T2. Hence the schedule is equivalent to the serial order T1, T2.

Precedence graph method

  1. Create one node for each transaction.
  2. Draw edge Ti → Tj if an operation of Ti conflicts with and occurs before an operation of Tj on the same item.
  3. If the graph is acyclic, the schedule is conflict serializable.

2) View serializability

A schedule is view serializable if it is view equivalent to some serial schedule. Two schedules are view equivalent if:

  • Initial reads are preserved,
  • Reads-from relationships are preserved, and
  • Final write on each data item is preserved.

Example of view-serializable but not conflict-serializable schedule

S2: W1(X)  W2(X)  W1(X)

This schedule contains a cycle in the conflict graph (T1 → T2 and T2 → T1), so it is not conflict serializable. However, since there are no reads and the final write is by T1, it is view equivalent to serial schedule T2, T1. Hence it is view serializable.

Comparison

AspectConflict SerializabilityView Serializability
BasisOrder of conflicting operationsView equivalence
TestingEasy using precedence graphMore complex
RelationshipSubsetSuperset
Conclusion: Serializability ensures correctness of concurrent execution. Conflict serializability is easier to test and widely used, while view serializability is more general but harder to verify.
Module 5 • Q3

Explain the Two-Phase Locking (2PL) protocol. Discuss its advantages and disadvantages.

Definition

Two-Phase Locking (2PL) is a concurrency control protocol that ensures conflict serializability by dividing transaction locking into two phases: a growing phase and a shrinking phase.

Phases of 2PL

Rules of 2PL

  1. Before reading a data item, transaction must acquire a shared (S) lock.
  2. Before writing a data item, transaction must acquire an exclusive (X) lock.
  3. Once a transaction releases any lock, it cannot acquire any new lock.

Phases

  1. Growing phase: transaction acquires locks and may upgrade them, but does not release any.
  2. Shrinking phase: transaction releases locks and cannot acquire new ones.

Strict 2PL

In Strict 2PL, all exclusive locks are held until commit or abort. This prevents cascading rollbacks and is widely used in practice.

Advantages

  • Guarantees conflict serializability.
  • Conceptually simple and practical.
  • Strict 2PL avoids cascading aborts.
  • Suitable for multi-user transaction processing.

Disadvantages

  • May lead to deadlocks.
  • Can reduce concurrency due to waiting.
  • Lock management overhead exists.

Applications

2PL is used in banking, reservation systems, inventory systems, and OLTP databases where correctness is more important than maximum parallelism.

Conclusion: Two-Phase Locking is a standard lock-based concurrency protocol. By separating lock acquisition and release into two phases, it guarantees serializability, though deadlock handling is necessary.
Module 5 • Q4

Explain various concurrency control techniques used in database systems with suitable examples.

Introduction

Concurrency control is the process of coordinating simultaneous execution of transactions so that database consistency is preserved and anomalies such as lost update, dirty read, and inconsistent retrieval are avoided.

Main techniques

  1. Lock-based protocols
    • Use shared and exclusive locks.
    • Example: 2PL and Strict 2PL.
    • Advantage: strong correctness guarantee.
    • Limitation: deadlocks may occur.
  2. Timestamp ordering protocol
    • Each transaction gets a timestamp.
    • Operations are executed in timestamp order.
    • If an operation violates timestamp order, transaction is rolled back.
    • Advantage: deadlock-free.
    • Limitation: more rollbacks may occur.
  3. Validation / Optimistic concurrency control
    • Assumes conflicts are rare.
    • Transaction phases: read, validate, write.
    • Used in low-conflict environments.
    • Advantage: high concurrency.
    • Limitation: validation failures under high contention.
  4. Multiversion concurrency control (MVCC)
    • Maintains multiple versions of data items.
    • Readers can access old committed versions without blocking writers.
    • Advantage: excellent read concurrency.
    • Limitation: version management overhead.
  5. Graph-based / tree-based protocols
    • Transactions follow a predefined partial order on data items.
    • Can prevent deadlocks under certain conditions.

Typical anomalies prevented

  • Lost update: one update overwrites another.
  • Dirty read: reading uncommitted data.
  • Unrepeatable read: same query returns different values in one transaction.
  • Phantom problem: row set changes between repeated predicate queries.

Comparison table

TechniqueMain ideaAdvantageDisadvantage
Lock-basedControl access with locksReliableDeadlocks
Timestamp orderingEnforce timestamp orderNo deadlocksMore restarts
OptimisticValidate at commit timeGood for low conflictsAbort on validation failure
MVCCKeep multiple versionsHigh read concurrencyStorage/version overhead
Conclusion: Different concurrency control techniques suit different workloads. Lock-based protocols are common in OLTP systems, timestamp and optimistic methods are useful in special cases, and MVCC is excellent for read-intensive environments.
Module 5 • Q5

Discuss database recovery techniques. Explain log-based recovery, deferred update, immediate update, and checkpointing.

Introduction

Database recovery is the process of restoring the database to a correct and consistent state after failures such as transaction failure, system crash, or media failure.

Need for recovery

  • To preserve atomicity and durability.
  • To recover committed data after crash.
  • To remove effects of incomplete transactions.

1) Log-based recovery

The DBMS maintains a log file containing records of transaction actions.

Typical log records:

<start T>
<T, X, old_value, new_value>
<commit T>
<abort T>

The log is written according to the Write-Ahead Logging (WAL) rule, that is, log records are written to stable storage before the corresponding database page is written.

2) Deferred update

In deferred update, changes made by a transaction are not written to the database until the transaction commits.

  • If crash occurs before commit, no undo is required.
  • After crash, only REDO of committed transactions may be needed.

3) Immediate update

In immediate update, changes may be written to the database before the transaction commits.

  • If crash occurs, uncommitted changes must be undone.
  • Committed changes may need redo if not yet reflected on disk.
  • Hence both UNDO and REDO may be required.

4) Checkpointing

A checkpoint is a recovery mechanism that periodically saves a known consistent state and marks it in the log.

Purpose: to reduce recovery time by limiting how far back the recovery manager must scan.

At checkpoint time:

  1. Modified buffers are flushed to disk.
  2. A checkpoint record is written into the log.
  3. During crash recovery, scanning can start from the latest checkpoint instead of the beginning of the log.
Recovery idea using log and checkpoint

Advantages of recovery techniques

  • Ensures durability of committed data.
  • Prevents permanent corruption due to crashes.
  • Improves reliability of transaction processing systems.

Limitations

  • Log maintenance introduces overhead.
  • Recovery design is complex.
  • Frequent checkpoints may affect performance.
Conclusion: Database recovery relies on logging, undo/redo mechanisms, deferred or immediate update strategies, and checkpointing. These techniques together ensure that the database can be restored to a consistent state after failures.
